# 回测框架开发计划

## 1. 项目概述

### 1.1 项目定位
这是一个基于Python的量化交易回测框架，设计上兼容掘金API，支持多频率数据和多种策略类型。框架采用模块化设计，核心组件包括上下文管理、账户管理、回测引擎和结果分析器。

### 1.2 核心组件

| 组件 | 文件 | 功能 |
|-----|------|------|
| **Context** | storage.py | 管理全局状态、数据缓存和订阅信息 |
| **AccountManager** | account.py | 处理订单、管理持仓和记录交易 |
| **Engine** | engine.py | 加载历史数据并按时间轴运行策略 |
| **AccountAnalyzer** | analysis.py | 分析回测结果，生成各种指标和HTML报告 |

### 1.3 工作流程

```
1. 初始化策略
   ↓
2. 订阅数据（context.subscribe）
   ↓
3. 加载历史数据（engine.add_data）
   ↓
4. 运行回测（engine.run）
   ├─ 按时间轴推进
   ├─ 调用策略的 on_bar 方法
   └─ 记录账户快照
   ↓
5. 分析结果（AccountAnalyzer）
   ↓
6. 生成HTML报告
```

## 2. 当前机制分析

### 2.1 数据初始化机制

当前采用手动加载方式：

```python
class 动量策略:
    def __init__(self):
        # 1. 先订阅
        context.subscribe(symbols=['SHSE.000300'], freq='1d', count=200)
        
        # 2. 手动调用外部API获取数据
        for symbol in symbols:
            df = history(symbol=symbol, frequency='1d', 
                        start_time='2019-01-01', end_time='2024-12-31', ...)
            
            # 3. 手动添加数据
            engine.add_data(symbol, '1d', df)
```

### 2.2 优点

- ✅ 参考掘金订阅机制，保持API兼容性
- ✅ 滚动缓存机制，与掘金一致
- ✅ 模块化设计，account可独立替换
- ✅ 回测/实盘统一，策略代码不变

### 2.3 问题

- ❌ 重复配置：subscribe 和 add_data 都要指定 symbol/freq
- ❌ 手动数据加载：每个策略都要写重复的数据加载代码
- ❌ 时间范围冗余：run(start_time, end_time) 和 history(start_time, end_time) 重复指定
- ❌ 缺乏自动化：用户需要手动管理数据源和时间范围

## 3. 方案对比

### 3.1 方案对比表

| 对比维度 | 当前方案（手动加载） | 方案1：自动数据加载 | 方案2：数据加载器模式 | 方案3：智能缓存机制 |
|---------|---------------------|-------------------|-------------------|-------------------|
| **代码简洁性** | ❌ 需要写大量重复代码 | ✅ 只需 subscribe | ✅ 只需 subscribe + 设置 loader | ✅ 只需 subscribe |
| **配置复杂度** | ❌ 需要配置3次（订阅、数据加载、引擎运行） | ✅ 只需配置2次（订阅、引擎运行） | ✅ 需配置3次（订阅、设置loader、引擎运行） | ✅ 只需配置2次（订阅、引擎运行） |
| **重复代码** | ❌ 每个策略都要写数据加载代码 | ✅ 零重复代码 | ✅ 零重复代码 | ✅ 零重复代码 |
| **时间范围管理** | ❌ 需要手动计算（start_time - 缓存长度） | ✅ 自动计算 | ✅ 自动计算 | ✅ 自动计算 |
| **数据源灵活性** | ✅ 可以自由切换数据源 | ⚠️ 需要修改引擎或传入参数 | ✅ 通过 DataLoader 接口灵活切换 | ✅ 通过 DataLoader 接口灵活切换 |
| **学习成本** | ❌ 需要理解数据加载流程 | ✅ 低，符合直觉 | ⚠️ 中等，需要理解 DataLoader | ✅ 低，符合直觉 |
| **性能优化** | ❌ 每次都加载全部数据 | ✅ 可以按需加载 | ✅ 可以按需加载 | ✅ 智能缓存，按需加载 |
| **错误处理** | ❌ 需要手动处理数据加载错误 | ✅ 引擎统一处理 | ✅ DataLoader 统一处理 | ✅ 引擎统一处理 |
| **多数据源支持** | ⚠️ 需要手动切换 API | ⚠️ 需要修改引擎代码 | ✅ 实现不同 DataLoader 即可 | ✅ 实现不同 DataLoader 即可 |
| **代码可维护性** | ❌ 数据加载逻辑分散 | ✅ 集中在引擎中 | ✅ 集中在 DataLoader 中 | ✅ 集中在引擎中 |
| **扩展性** | ❌ 添加新数据源需要修改策略 | ⚠️ 需要修改引擎 | ✅ 只需新增 DataLoader | ✅ 只需新增 DataLoader |
| **向后兼容** | ✅ 当前方案 | ❌ 需要重构 | ⚠️ 可以保留旧接口 | ❌ 需要重构 |

### 3.2 推荐方案

**最佳方案：方案2 + 方案3 的组合**

结合两者的优点：
- 使用 **DataLoader 接口**（方案2）实现数据源灵活性
- 在 **Engine 中实现智能缓存**（方案3）实现性能优化

## 4. 实盘对接路径

### 4.1 路径1：account 模块独立

```
┌─────────────┐
│  策略层      │  (统一)
├─────────────┤
│  Context     │  (统一)
├─────────────┤
│  Account     │  ← 可替换：掘金/QMT/自定义
├─────────────┤
│  数据源      │  ← 回测：手动加载 / 实盘：API推送
└─────────────┘
```

**优势**：
- 策略代码完全不变
- 可以直接对接掘金实盘
- 也可以对接QMT等其他平台

### 4.2 路径2：加入API数据驱动

```
┌─────────────┐
│  策略层      │
├─────────────┤
│  Context     │  ← 增加 API 数据驱动
├─────────────┤
│  Engine      │  ← 回测引擎
│  LiveEngine  │  ← 实盘引擎（新增）
├─────────────┤
│  Account     │
└─────────────┘
```

**优势**：
- 框架更完整
- 统一的数据接口
- 便于切换回测/实盘

## 5. 最终决策

### 5.1 核心决策

**聚焦做好回测框架，account模块独立化，后期接入PostgreSQL**

### 5.2 理由

1. **回测框架是核心** - 实盘对接需要构建数据驱动环节，复杂度高
2. **account模块独立** - 便于后期对接掘金/QMT等实盘平台
3. **PostgreSQL支持** - 实现数据持久化，支持大规模回测和数据分析

## 6. Account模块独立性改进方案

### 6.1 架构设计

```
┌─────────────────────────────────────┐
│      AccountManager (业务逻辑)      │
│  - order_volume()                   │
│  - order_percent()                  │
│  - get_account()                    │
│  - get_position()                   │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│      Storage (存储抽象接口)         │
│  - save_trade_log()                 │
│  - save_snapshot()                  │
│  - load_trade_log()                 │
│  - load_snapshot()                  │
└─────────────────────────────────────┘
         ↓              ↓              ↓
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ MemoryStore │ │ PGStore     │ │ CSVStore    │
│ (当前)      │ │ (PostgreSQL)│ │ (CSV文件)   │
└─────────────┘ └─────────────┘ └─────────────┘
```

### 6.2 存储抽象接口

```python
class Storage(ABC):
    """存储抽象接口"""
    
    @abstractmethod
    def save_trade_log(self, trade_log: List):
        """保存交易记录"""
        pass
    
    @abstractmethod
    def save_snapshot(self, snapshot):
        """保存账户快照"""
        pass
    
    @abstractmethod
    def load_trade_log(self, start_time: datetime = None, end_time: datetime = None) -> List:
        """加载交易记录"""
        pass
    
    @abstractmethod
    def load_snapshots(self, start_time: datetime = None, end_time: datetime = None) -> List:
        """加载账户快照"""
        pass
    
    @abstractmethod
    def clear(self):
        """清空存储"""
        pass
```

### 6.3 存储实现

#### 6.3.1 MemoryStore（内存存储）
- 保留当前实现
- 适用于小规模回测
- 数据仅存储在内存中

#### 6.3.2 PGStore（PostgreSQL存储）
- 支持数据持久化
- 数据表结构：
  - `trade_logs` - 交易记录表
  - `account_snapshots` - 账户快照表
  - `position_snapshots` - 持仓快照表
- 支持时间范围查询
- 支持批量插入优化

#### 6.3.3 CSVStore（CSV文件存储）
- 可选实现
- 便于数据导出和共享
- 支持Excel兼容格式

### 6.4 AccountManager改造

```python
class AccountManager:
    def __init__(
        self,
        init_cash: float = 1e6,
        fee_config: Dict = None,
        storage: Storage = None,
    ):
        """
        :param init_cash: 初始资金
        :param fee_config: 手续费配置
        :param storage: 存储接口（默认使用内存存储）
        """
        self.cash = round(init_cash, 2)
        self.positions: Dict[str, Dict] = {}
        self.fee_config = fee_config or {...}
        
        # 使用存储接口
        self.storage = storage or MemoryStore()
```

### 6.5 使用示例

```python
# 使用内存存储（默认）
account = AccountManager(init_cash=1e6)

# 使用PostgreSQL存储
from ft2.account import PGStore
pg_store = PGStore(
    dbname='backtest',
    user='postgres',
    password='password',
    host='localhost'
)
account = AccountManager(init_cash=1e6, storage=pg_store)
```

## 11. 总结

本开发计划聚焦于将回测框架的 account 模块独立化，通过引入存储抽象层，实现数据持久化和灵活切换。主要改进包括：

1. **存储抽象层** - 统一的存储接口，支持多种存储后端
2. **PostgreSQL支持** - 实现数据持久化，支持大规模回测
3. **向后兼容** - 默认使用内存存储，保持现有代码不变
4. **可扩展性** - 通过实现 Storage 接口即可扩展新的存储后端

通过这些改进，框架将具备更好的可维护性、扩展性和实用性，为后续对接实盘平台奠定基础。
